# Contributing to EU DECIDO

First of all, thank you for your interest in contributing to EU DECIDO.

This project is an experimental initiative exploring digital democracy, citizen participation, transparent governance, and responsible AI using open technologies.

We welcome:
- developers,
- researchers,
- designers,
- governance specialists,
- and contributors interested in civic technology and democratic innovation.

---

# Before Contributing

Please:
- read the README,
- explore existing Issues and Discussions,
- and understand the experimental nature of the project.

At this stage, architecture and governance are still evolving.

---

# Ways to Contribute

You can contribute through:
- bug fixes,
- feature suggestions,
- UI/UX improvements,
- documentation,
- AI agent architecture,
- backend systems,
- mobile development,
- security reviews,
- or governance proposals.

Constructive criticism and technical feedback are also highly valuable.

---

# Contribution Workflow

## 1. Fork the repository

Create your own fork of the project.

---

## 2. Create a branch

Please create a dedicated branch for your changes.

Example:

```bash
git checkout -b feature/forum-improvements

---

# 3. Commit Your Changes

Use clear and descriptive commit messages.

Examples:

```bash
Add anonymous voting validation
```

```bash
Fix proposal synchronization issue
```

---

# 4. Open a Pull Request

Submit a Pull Request with:
- a clear explanation,
- screenshots if relevant,
- and reasoning behind the proposed changes.

All Pull Requests are reviewed before merging.

---

# Pull Request Guidelines

Please:
- keep Pull Requests focused,
- avoid unrelated changes,
- and explain architectural decisions when relevant.

Large or sensitive changes should be discussed first through Issues or Discussions.

---

# Branching Strategy

Current branch structure:

- `main` → stable branch
- feature branches → active development

Examples:

```bash
feature/elections-system
```

```bash
feature/ai-orchestrator
```

```bash
fix/authentication
```

```bash
docs/readme-update
```

Direct commits to `main` may be restricted for external contributors.

---

# Coding Style

## General Principles

We value:
- readability,
- modularity,
- maintainability,
- and transparency.

Avoid unnecessary complexity.

---

## Kotlin / Mobile

- Prefer clean architecture principles
- Keep UI components modular
- Avoid tightly coupled business logic
- Favor shared multiplatform code when possible

---

## Backend / Python

- Follow FastAPI best practices
- Use type hints where possible
- Keep routes and services separated
- Avoid business logic inside controllers

---

## AI Systems

Future AI systems should prioritize:
- transparency,
- auditability,
- modular orchestration,
- and Responsible AI principles.

---

# Security

If you discover security vulnerabilities or sensitive issues, please avoid publishing them publicly in Issues.

Instead, contact the maintainers directly.

---

# Discussions & Governance

This project includes technical, ethical and governance dimensions.

Contributors are encouraged to:
- participate respectfully,
- discuss ideas constructively,
- and help improve the platform collaboratively.
